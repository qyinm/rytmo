Ideation: Pace Surprise-Me Directions

Pace has enough pieces to stop being described as a Pomodoro timer with music. The strongest directions treat it as a private Mac-native rhythm layer: it compiles calendar, tasks, music, focus state, and recovery into a live workday surface.

Grounding Context

Codebase Context. Pace is a native macOS Swift 6 / SwiftUI app with Pomodoro timer, YouTube music, SwiftData tasks/playlists/focus sessions, calendar integrations, Firebase/GoogleSignIn auth, Amplitude, Sparkle updates, custom notch/glance UI, dashboard, and a Next.js landing page.

The repo shows a product identity gap: the README still frames Pace as a menu bar Pomodoro + YouTube app, while the code and changelog show calendar, tasks, dashboard, notch UI, focus stats, settings, and update infrastructure.

The heaviest local complexity is in two areas. Calendar has EventKit + Google Calendar, scope upgrades, token refresh, caches, periodic sync, disconnect cleanup, and a 5000-event cap. Release has Sparkle ZIP/appcast/DMG generation, GitHub release support, unsigned archive fallback, post-signing, and recent rename fallout.

Past learnings. The repo already learned that Xcode archive creation and distributable app signing should be separated, and that product renames drift across app targets, packages, entitlements, release artifacts, landing page constants, repo/domain state, and Sparkle URLs unless coordinated.

External context. Endel, Brain.fm, Portal, Rize, Tiimo, vashTimer, RISE, and current macOS agent benchmarks point toward adaptive sound, passive time intelligence, neuro-inclusive planning, accessible timekeeping, and agent-readable native UI as live opportunity spaces.

1Idea 1. Pace Compiler / Autopace Workday

Description: Turn messy inputs — tasks, calendar constraints, focus history, music preference, skips, and interruptions — into an executable day rhythm. The timer becomes the runtime, not the product: Pace proposes warm-up blocks, deep-work peaks, shallow-work lanes, cooldowns, recovery windows, and music intensity shifts as the day changes.

Basis

direct: Pace already has timers, tasks, calendar, music, persisted focus sessions, skip behavior, and a 24-hour timeline promise. external: Tiimo supports visual planning and executive-function scaffolding; RISE maps adaptive music intensity to activity phases.

Rationale

This is the best product-thesis candidate because it explains why all the existing subsystems belong together. It shifts Pace from “start a timer” to “let the Mac help decide what kind of time you should be in next.”

Downsides

High product and UX risk: bad recommendations would feel presumptuous. It needs a strong fallback where users can override, correct, or keep manual timer behavior.

2Idea 2. Focus Session Ledger / Focus Graph

Description: Build a canonical local semantic model for Pace: planned intent, actual start/end, skipped segments, interruption reason, music context, linked task, linked calendar event, and current focus state. This becomes the shared source for dashboard history, notch decisions, passive receipts, analytics, agent-readable state, and privacy controls.

Basis

direct: TimerManager already persists focus sessions and skip behavior; Pace has tasks, calendar, music, focus stats, and notch UI. external: Rize shows private passive time intelligence can work without screenshots or keylogging.

Rationale

This is the strongest foundation move. Without a ledger/graph, every smart feature will create its own partial truth about what the user is doing.

Downsides

It is mostly invisible until paired with a user-facing surface. Scope can balloon if the graph tries to model everything before proving one workflow.

Confidence 90%Complexity Medium

3Idea 3. No Dashboard: The Notch Is the Product

Description: Flip the center of gravity: the daily product is never a dashboard. The custom notch/glance surface becomes the command center for the current task, next calendar conflict, timer state, and music intensity; the dashboard becomes archive, configuration, and history.

Basis

direct: Pace already has custom notch/glance UI, dashboard navigation, timer, calendar, tasks, music, and focus sessions. external: Portal shows productivity software can be sold as presence and environment rather than tracking panels.

Rationale

This sharpens the native Mac advantage. Pace can be lighter than the work it supports instead of becoming another productivity hub the user has to manage.

Downsides

The notch surface has limited space and high interaction risk. It needs ruthless prioritization and accessibility work so it does not become a cramped novelty.

Confidence 80%Complexity Medium

4Idea 4. Adaptive Sound / Cognitive Gearbox

Description: Treat YouTube/music as the focus control surface, not entertainment. Pace maps warm-up, ramp, deep work, urgent push, cooldown, and recovery modes to sound and pacing, using calendar pressure, task type, time of day, and skip patterns.

Basis

direct: Pace already combines YouTube music, playlists/tracks, timer state, focus sessions, calendar, and tasks. external: Endel adapts soundscapes to context; Brain.fm and Focus@Will position functional music around attention; RISE adapts music intensity to activity phases.

Rationale

This makes “Focus with Rhythm” literal. Music becomes what helps the user shift gears, not a playlist embedded beside a timer.

Downsides

Credibility matters: focus-audio claims can sound pseudoscientific. YouTube control granularity may limit how adaptive the first version can be.

Confidence 78%Complexity High

5Idea 5. Rhythm Debt / Work Readiness

Description: Replace vanity productivity stats with a humane readiness model. Pace tracks mismatches between planned intensity, actual attention, recovery, meeting load, skipped sessions, and late-day overload, then tells the user when they are borrowing focus from tomorrow.

Basis

direct: TimerManager persists completed and skipped focus sessions; calendar load and focus stats already exist as raw material. external: Endel’s circadian/ultradian adaptation and Tiimo’s neuro-inclusive framing both support time-quality over output-count thinking.

Rationale

The metric fits the brand: rhythm, not hustle. It also gives skips and failed sessions positive diagnostic value instead of hiding them as bad outcomes.

Downsides

It can become hand-wavy if not anchored in transparent inputs. The app must avoid moralizing user behavior.

Confidence 76%Complexity Medium

6Idea 6. Agent-Readable Pace

Description: Design Pace as a native Mac app that humans and local agents can both understand. The notch, timer, tasks, calendar state, focus sessions, and settings expose accessibility labels, deterministic commands, inspectable state, and semantic actions such as “current focus,” “next session,” and “protect this block.”

Basis

direct: Pace is a native SwiftUI macOS app with custom notch/glance UI and productivity state worth exposing. external: MacArena, MacAgentBench, and Screen2AX point toward agent-readable and accessibility-rich macOS UI as a strategic frontier.

Rationale

This links accessibility, QA, automation, and product differentiation. If local agents increasingly operate Macs, Pace can become a focus boundary they can respect without screen scraping.

Downsides

The value is partly future-facing. It needs near-term wins through accessibility, keyboard control, and testability to justify itself before agent workflows mature.

Confidence 82%Complexity Medium

7Idea 7. Release Cockpit / Rename-Safe Shipping

Description: Build a preflight cockpit for native distribution and product identity. It verifies signing, Sparkle appcast, DMG/ZIP assets, GitHub release assets, landing-page download constants, version names, rollback readiness, and narrative drift before distribution.

Basis

direct: The repo already has Sparkle zip/appcast/DMG generation, GitHub release support, unsigned archive fallback, post-signing, and docs capturing release-signing and rename drift failures.

Rationale

This is developer-facing, but it compounds. Every future product bet is easier if shipping is boring, auditable, and rename-safe.

Downsides

It does not improve the end-user experience directly. It should stay small and diagnostic before becoming a large internal platform.

Confidence 88%Complexity Medium
